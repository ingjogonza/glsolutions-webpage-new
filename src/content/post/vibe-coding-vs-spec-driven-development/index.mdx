## ¿Qué es vibe coding?

Vibe coding es cuando alguien — humano o IA — escribe código guiándose por lo que "se siente bien" en el momento, sin un documento previo que defina qué debe hacer el sistema. Funciona perfecto para un prototipo de fin de semana. Falla en cuanto el sistema tiene que sobrevivir a un segundo desarrollador, un inversor técnico o un regulador.

## El problema no es la IA — es la ausencia de mapa

La IA generativa escribe código increíblemente rápido. El problema es que código rápido sin un mapa previo es código que nadie — ni la persona que lo pidió — puede explicar seis meses después. La IA no entiende tu negocio; ejecuta instrucciones. Si esas instrucciones nunca se escribieron como spec, lo que queda es la interpretación de la IA del momento, no una decisión de arquitectura.

## Qué es Spec Driven Development

SDD invierte el orden: primero se documenta qué debe hacer el sistema, en qué casos, y por qué — incluyendo los casos borde y los errores posibles. Recién cuando eso está acordado, la IA construye siguiendo ese mapa. Las decisiones de arquitectura las siguen tomando los ingenieros, no la herramienta.

## Comparación directa

| | Vibe coding | Spec Driven Development |
|---|---|---|
| Velocidad inicial | Muy alta | Media — hay un paso de documentación antes |
| Documentación | Ninguna o mínima | Completa, aprobada antes del código |
| Onboarding de un nuevo dev | Semanas, dependiente de quien construyó | Horas, basado en documentos |
| Auditoría / compliance | Difícil o imposible | Cada decisión queda registrada |
| Deuda técnica | Invisible hasta que explota | Visible y gestionada desde el inicio |
| Costo real a 6 meses | Alto — reescritura o parálisis | Predecible — el precio se fijó con el alcance |

## Cuándo vibe coding sí tiene sentido

Para un prototipo descartable, una validación de mercado de unos días, o una demo que nunca va a producción, vibe coding es la herramienta correcta. El error es usar el mismo enfoque para un sistema que va a manejar datos de usuarios, dinero, o decisiones regulatorias.

## Cuándo necesitas SDD

Si tu startup maneja datos sensibles o dinero, está levantando capital y necesita mostrar arquitectura a un inversor técnico, o planea contratar equipo técnico propio en los próximos meses, el costo de no documentar es mayor que el costo de documentar.

## La pregunta que realmente importa

No es "¿cuál es más rápido?". Es "¿quién va a entender este sistema en seis meses, y qué pasa si esa persona no soy yo?". Si no tienes una respuesta clara, probablemente necesitas un mapa antes del código.
